Maybe 使 ECMAScript 更安全

2020-01-11

ECMAScript 為動態語言，Function Parameter 並不必指定 Type Annotation，因此可以傳入任何 Type，理論上必須在 Runtime 使用 typeof 做 Type Check，否則依賴 Type Coercion 很容易產生 Bug；Maybe 提供了另外一種方式：只會將正確 Type 進行運算，而不需使用 typeof 檢查。

Version

macOS Mojave 10.15.2
VS Code 1.41.1
Quokka 1.0.271
Crocks 0.12.4

Number

let inc = x => x + 1

let fn = n => inc(n)

fn(2) // ?

很簡單的 inc()，當輸入為 number 2 時，毫無懸念結果為 3。

String

let inc = x => x + 1

let fn = n => inc(n)

fn('8') // ?

但傳入改為 string 8 時，結果為 81，且 81 為 string，並不是預期的 number。

由於 ECMAScript 為動態語言，因此可以傳入任何 type，而當 + 遇到 string 時，會將兩個 operand 都轉成 string，+ 從原本的 add() 變成 concat()，因此最後結果為 string 81

Undefined

let inc = x => x + 1

let fn = n => inc(n)

fn(undefined) // ?

當傳入為 undefined 時，結果為 NaN。

81 與 NaN 都不是我們預期結果，要避免這些情形發生，唯一的方法就是確認輸入只能是 number 才能使用 inc() 運算，其他 type 都不執行 inc()

typeof

let inc = x => x + 1

let fn = n => typeof n === 'number' ? inc(n) : 0

fn(2) // ?
fn('8') // ?
fn(undefined) // ?

為了確保輸入一定是 number，我們當然可以在 fn() 加上 typeof 判斷，確定是 number 才執行 inc()，否則傳回 default 值 0。

如此就不會是 string 或 NaN 這些不是我們預期結果。

但這種寫法只能算是 short term solution：

  • 每個 function 都必須檢查其 parameter type，弄髒了原本邏輯
  • 無法在一般情況就發現錯誤
  • 必須考驗 coder 細心程度，只要不夠細心，unit test 一樣測不到

Wrap into Maybe

import { Maybe, map } from 'crocks'

let { Just } = Maybe

let inc = x => x + 1

let fn = map(inc)

fn(Just(2)) // ?

引進了 Maybe，它包含兩種 type：Just 與 Nothing：

Maybe a = Nothing | Just a
  • Just：我們所預期 type，將會執行運算
  • Nothing：我們不預期 type，不會執行運算

以本例而言，我們預期 type 是 number，是故為 Just；而 string 與 undefined 都不是我們預期 type，是故為 Nothing。

第 1 行

import { Maybe, map } from 'crocks'

由 crocks import 進 Maybe object，另外也從 crocks import 進 map()，此與 Ramda 的 map() 完全相同。

第 3 行

let { Just } = Maybe

從 Maybe destructure 出 Just()，將來不必再使用 Maybe.Just()。

第 7 行

let fn = map(inc)

map() 如同 Ramda 的 map() 一樣，data 也是最後一個 argument，因此可 point-free 省略。

當 map() 用在 Maybe 時，它會幫我們將 Maybe 內的 value 透過傳入 function 改變之，最後再回傳 Maybe。

由於 fn() 回傳為 Maybe，因此印出 Just 3，而非原本的 3。

目前我們已經將 number 包進 Maybe，但印出也是 Maybe，而非原本的 number ，稍後會從 Maybe 萃取出 number

Just

import { Maybe, map } from 'crocks'

let { Just } = Maybe

let inc = x => x + 1

let fn = map(x => console.log('calling inc()') || inc(x))

fn(Just(2)) // ?

我們雖然將 inc() 傳入 map()，但 inc() 真的有被執行嗎 ?

第 7 行

let fn = map(x => console.log('calling inc()') || inc(x))

特別在傳入 map() 的 function 加上 console.log()，測試 inc() 是否有被執行。

由於 console.log() 會回傳 undefined，視為 falsy value，因此一定會執行 || 右側的 inc(x)。

console.log() 配合 || 為 ECMAScript 常用測試 callback 手法

印出了 calling inc()，表示若輸入為 Just， 真的有執行傳入 map() 的 function。

Nothing

import { Maybe, map } from 'crocks'

let { Nothing } = Maybe

let inc = x => x + 1

let fn = map(x => console.log('calling inc()') || inc(x))

fn(Nothing()) // ?

inc() 與 fn() 完全不變，只是改將 Nothing() 傳入 fn()。

沒顯示 calling inc()，表示 inc() 根本沒執行，直接回傳 Nothing。

只有 Just 才會執行 map()，Nothing 完全不執行，這確保了只有正確 type 才會執行 inc()，不需我們自己做 typeof 檢查，也不會產生不預期結果

safeNum()

import { Maybe, map, compose } from 'crocks'

let { Just, Nothing } = Maybe

let inc = x => x + 1

let safeNum = v => typeof v === 'number' ? Just(v) : Nothing()

let fn = compose(map(inc), safeNum)

fn(2) // ?
fn('8') // ?
fn(undefined) // ?

第 9 行

let fn = compose(map(inc), safeNum)

由 Just 與 Nothing 我們可發現，只要我們能將 value 包進 Maybe，儘管不做 typeof 檢查，也能確保沒有不預期結果，所以我們需要 safeNum() 將任何 value 包進 Maybe。

將 safeNum() 與 map(inc) 加以組合成最後 fn()。

第 7 行

let safeNum = v => typeof v === 'number' ? Just(v) : Nothing();

使用 typeof 判斷是否為 number，若是 number，則包成 Just，否則包成 Nothing。

Ｑ：到底還是使用了 typeof ?

底層雖然會使用 typeof，但即將包成 higher order function，請繼續看下去。

Higher Order Function

import { Maybe, map, compose } from 'crocks'

let { Just, Nothing } = Maybe

let inc = x => x + 1

let isNumber = x => typeof x === 'number'

let safe = fn => x => fn(x) ? Just(x) : Nothing()

let fn = compose(map(inc), safe(isNumber))

fn(2) // ?
fn('8') // ?
fn(undefined) // ?

第 7 行

let isNumber = x => typeof x === 'number'

將 number 寫死在 safeNum() 當然不好，可將此部分抽成 isNumber()，將來可透過 function compostion 由 safe(isNumber) 組合出 safeNum()，重複使用程度更高。

第 9 行

let safe = fn => x => fn(x) ? Just(x) : Nothing()

將 safeNum() 重構成更一般性的 safe() higher order function，可透過傳入 function 組合出能將任何 value 包進 Maybe 的 function。

11 行

let fn = compose(map(inc), safe(isNumber))

由 safe(isNumber) 組合出 safeNum()，最後依然使用 compose() 組合，如此可安全將任何 value 包進 Maybe。

Helper Function

import { inc, map, compose } from 'ramda'
import { safe, isNumber } from 'crocks'

let fn = compose(map(inc), safe(isNumber))

fn(2) // ?
fn('8') // ?
fn(undefined) // ?

事實上 Crocks 已經提供了 safe() 與 isNumber()，可直接使用。

連 inc() 也一併使用 Ramda 版本。

目前只剩下最後一哩路： fn() 回傳 Maybe，所以印出結果是錯的，不過也因為能在一般狀況下就發現錯誤，我們很容易修正

Unwrap from Maybe

import { inc, map, compose } from 'ramda'
import { safe, isNumber, option } from 'crocks'

let fn = compose(option(0), map(inc), safe(isNumber))

fn(2) // ?
fn('8') // ?
fn(undefined) // ?

Crocks 提供了 options()，讓我們提供當 Maybe 為 Nothing 時該回傳的 default value，因為 Nothing 完全不會經過 inc() 運算。

可直接將 option() 組合其他 function。

fn() 的 signature 完全沒變，結果也完全相同。

Summary

比較這兩種寫法，我們得到了什麼 ?

Imperative

let fn = v => typeof v === 'number' ? inc(v) : 0

我們必須時時刻刻想著要使用 typeof 檢查，但儘管粗心沒檢查，一般情況 也不會錯，但特殊狀況就會錯，這就是 bug 來源。

Functional

let fn = compose(option(0), map(inc), safe(isNumber))

若使用 FP 的 Maybe，思維就不一樣了：

  • 先組合出能包進 Maybe 的 function，因此會先寫出 safe(isNumber)，就算你很粗心也被逼的要寫出來，否則無法繼續
  • 接下來要使 Maybe 能經過運算，一定得使用 map() 傳入 function
  • 若很粗心忘記寫 option()，一般情況顯示 Maybe 就是錯的，一定會回來補上 option()

也就是每個步驟都被逼著走，且在 一般情況 就會發現錯誤，不像 imperative 的 typeof 在一般情況不會錯，只有在 特殊情況 才會出錯，這就是 bug 來源。

Maybe 還可讓你在寫 unit test 時不用想很奇怪的 test case，因為 Maybe 會確保沒有不預期結果。

Conclusion

  • Maybe 事實上就是 FP 的 Monad，但不知道 Monad 也沒關係
  • Maybe 在 ECMAScript 的重要性遠超過其他語言，可避免動態語言產生不預期結果，大幅降低 bug 產生，使 ECMAScript 更安全
  • Crocks 已經提供了 Maybe 所需要的 helper function，可直接使用
