使用回傳 Maybe 的 find() 更安全

2020-04-14

Ramda 有些 Function 回傳 undefined，最典型的就是 find()，這使的 Happy Path 下很容易不小心就使用 undefined 而造成 Runtime Error，實務上建議改用 Crocks 回傳 Maybe 的 find() 取代。

ECMAScript

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => a => a.find(x => x.title === s).price

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

ECMAScript 的 find() 在 happy path 正常，若找不到資料會回傳 undefined，一不小心會造成 runtime error。

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => a => {
  let result = a.find(x => x.title === s) || { price: 0 }
  return result.price
}

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

由於 ECMAScript 的 undefined 設計，它就是要你用 if() 與 falsy value 在 runtime 判斷。

這樣雖然可行，但必須非常小心，因為 happy path 無法發現問題，實務上必須搭配 unit test 與 test case 才容易發，且商業邏輯還必須混雜 undefined 判斷

Ramda

import { find, either, pipe, prop, propEq } from 'ramda'

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => pipe(
  find(propEq('title')(s)),
  prop('price'),
)

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

Ramda 的 find() 雖然也會回傳 undefined，但 Ramda 哲學是 garbage in garbage out，因此 undefined 進了 prop() 不會產生 runtime error，只是也回傳 undefined。

雖然 Ramda 不會產生 runtime error 這點比 ECMAScript 優，但 happy path 仍然無法發現問題，必須在特定 test case 才會出現非預期的 undefined

import { find, either, pipe, prop, propEq, always } from 'ramda'

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => pipe(
  either(
    find(propEq('title')(s)),
    always({ price: 0 })
  ),
  prop('price'),
)

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

10 行

either(
  find(propEq('title')(s)),
  always({ price: 0 })
),

若要處理 find() 回傳的 undefined，要使用 either() 將 find() 包一層，若 undefined 則回傳 default value，如此 prop() 也能抓到。

either() 雖然可行，但總覺得不是那麼優雅，且還是有判斷 undefined 味道，只是改用 function 而已

Crocks

import { pipe, prop, propEq } from 'ramda'
import { find, option } from 'crocks'

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => pipe(
  find(propEq('title')(s)),
  prop('price')
)

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

Crocks 亦提供 find()，功能與 ECMAScript 與 Ramda 完全相同，但回傳 Maybe。

也因為是回傳 Maybe，儘管 happy path 一樣只能得到 undefined，此時就會提醒我們必須取得 Maybe 內部值。

import { pipe, prop, propEq } from 'ramda'
import { find, option } from 'crocks'

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => pipe(
  find(propEq('title')(s)),
  option({ price: 0 }),
  prop('price')
)

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

Maybe 類似 Promise 一樣，要取得內部值需透過其他 function，Promise 要透過 andThen()，而 Maybe 要透過 option()，且 option() 還必須為 Nothing 提供 default value。

如此只要 function pipeline 一層就可解決，也不需要特定 test case 發現問題。

import { pipe, propEq } from 'ramda'
import { find, option, chain, prop } from 'crocks'

let data = [
  { title: 'FP in JavaScript', price: 100 },
  { title: 'RxJS in Action', price: 200 },
  { title: 'Speaking JavaScript', price: 300 }
]

let f = s => pipe(
  find(propEq('title')(s)),
  chain(prop('price')),
  option(0),
)

f('Speaking JavaScript')(data) // ?
f('RxJS')(data) // ?

之前寫法雖然用到 Maybe，也不須任何判斷，但仍不是 Maybe 最佳寫法，很類似使用 Promise 時很急著使用 await 從 Promise 取出一樣。

Maybe 比較好的寫法是先使用 map() 或 chain() 直接對 Maybe 內部值處理，一直使用 Maybe 包著，最後再使用 option() 取出。

因為要使用 prop() 取出 property，這裡改用 Crocks 會回傳 Maybe 版本的 prop()。

也因為 prop() 會回傳 Maybe，因此要使用 chain() 避免兩層 Maybe。

最後使用 option() 提供 Nothing 的 default value 並從 Maybe 取出。

Q：不用判斷 find() 回傳 Nothing 嗎 ?

若 find() 回傳 Nothing，則不會執行 chain() 內的 prop()，而是直接下一步 option(0)，這就是 Functor / Monad 的精妙之處，也才是 FP 的正確寫法。

Conclusion

  • Crocks 的 find() 遠比 Ramda 的 find() 清爽，回傳 undefined 的 find() 還必須透過 either() 包一層，但回傳 Maybe 的 find() 只需組合 option() 即可，觀念類似 Promise
  • 回傳 Maybe 最重要的意義在於 happy path 就能發現問題，不必透過特殊 test case
  • 實務上建議只要 Ramda 回傳 undefined 的 function，都改用 Crocks 回傳 Maybe 版本
  • Functor / Monad 的精妙之處就是 null、undefined、empty value、exception 這類值都不會執行 map() 或 chain() 內的 function，因此不必做任何判斷，可直接以 happy path 使用 Just，最後再處理 Nothing 即可，也就是所謂 Railway Oriented Programming
