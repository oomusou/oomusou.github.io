深入探討 Class 與 Prototype 黑魔法

2019-12-25

ECMAScript 2015 雖然支援了 class 語法，但本質上仍是使用 Prototype 實作，本文深入探討其背後黑魔法。

Version

macOS Catalina 10.15.2
VS Code 1.41.1
Quokka 1.0.267
ECMAScript 2015

Factory Function

let createBook = (title, price) => ({
  title,
  price,    
})

let book = createBook('FP in JavaScript', 100)

let book_ = {
  showDescription: function() {
    return `${ this.title } : ${ this.price }`
  }
};
Object.setPrototypeOf(book, book_)

book.__proto__ // ?
book.showDescription() // ?

我們知道若將 method 放在 object，當建立多個 object 時，會不斷建立新的 method，較浪費記憶體，比較好的方式是將 property 建立在 object 內，而將 method 建立在 object 的 prototype。

第 1 行

let createBook = (title, price) => ({
  title,
  price,    
})

let book = createBook('FP in JavaScript', 100)

建立 createBook() factory function，只負責建立 object 的 property 即可。

第 8 行

let book_ = {
  showDescription: function() {
    return `${ this.title } : ${ this.price }`
  }
};
Object.setPrototypeOf(book, book_)

使用 object literal 直接建立 book_ 為 prototype object。

使用 Object.setPrototypeOf() 動態將 book_ 成為 book 的 prototype。

Object.setPropertyOf() 為 ES6 所新增

17 行

book.__proto__ // ?

使用 book.__proto__ 觀察其 prototype 為 book_。

__proto__ 為 ES6 所新增

Constructor Function

let Book = function(title, price) {
  this.title = title
  this.price = price
}

Book.prototype.showDescription = function() {
  return `${ this.title } / ${ this.price }`
}

let book = new Book('FP in JavaScript', 100)
book.__proto__ === Book.prototype // ?
book.showDescription() // ?

自己使用 factory function 實踐 prototype 雖然可行，但程式碼稍嫌冗長，ECMAScript 另外提供了 constructor function，讓我們以更精簡的方式使用 prototype。

第 1 行

let Book = function(title, price) {
  this.title = title
  this.price = price
}

類似 factory function 傳入 title 與 price，只負責建立 object 的 property 即可。

其中 this 為稍後 new 所建立的 object，因此可以直接使用 this.title 新增 property。

Constructor function 習慣會使用 PascalCase，有別於一般 function 使用 camelCase

第 6 行

Book.prototype.showDescription = function() {
  return `${ this.title } / ${ this.price }`
}

Constructor function 本質還是 function，而 function 自帶 prototype property 為 {}，直接對 {} 動態加上 fullName() method。

是否覺得這裡好像哪裡怪怪的 ? 稍後會說明

prototype 為 function 專屬的 property，但並不代表 function 的 prototype 是 {}。

可發現 Book 的 prototype 與 __proto__ 並不相同。

第 10 行

let book = new Book('FP in JavaScript', 100)

當 function 搭配 new 時，就搖身一變成為 constructor function，this 為所建立的 object。

11 行

book.__proto__ === Book.prototype // ?
book.showDescription() // ?

我們可發現一個很有趣的現象：book.__proto 與 Book.prototype 竟然指向相同 object，稍後會詳細說明。

let Book = function(title, price) {
  this.title = title
  this.price = price
}

Book.prototype = {
  constructor: Book,
  showDescription: function() {
    return `${ this.title } / ${ this.price }`
  }
}

let book = new Book('FP in JavaScript', 100)
book.__proto__ === Book.prototype // ?
book.showDescription() // ?

第 6 行

Book.prototype = {
  constructor: Book,
  showDescription: function() {
    return `${ this.title } / ${ this.price }`
  }
}

若 method 多時，其實有另外一種寫法：直接使用 object literal 方式定義 object，然後指定給 prototype。

不過這種寫法有個地方要小心，之前 Book.prototype.showDescription 是對 {} 動態新增 method，原本 {} 的 property 都還留著，其中一個最重要的 property 就是 constructor 指向 Book() constructor function。

但若將整個 object literal 指定給 Book.prototype 時，別忘了要自行補上 constructor: Book，這才符合原本 Book.prototype.constructor 要指向 constructor function 要求。

我在學習 prototype 時，最大卡關是觀念上明明應該對 object 的 prototype 新增 method，為什麼到最後是對 constructor function 的 prototype property 新增 method 呢 ?

其實剛剛有發現一個有趣的現象：

book.__proto__ === Book.prototype // ?

book.__proto__ 與 Book.prototype 指向同一個 object ！！

let Book = function(title, price) {
  this.title = title
  this.price = price
}

Book.prototype.showDescription = function() {
  return `${ this.title } / ${ this.price }`
}

let book = {}
Book.call(book, 'FP in JavaScript', 100)
book.__proto__ = Book.prototype

book.showDescription() // ?

10 行

let book = {}
Book.call(book, 'FP in JavaScript', 100)
book.__proto__ = Book.prototype

new 的黑魔法就是這 3 行，除了將 book 傳進 Book.call() 外，還將 Book.prototype 的記憶體位置複製給 book.__proto__，兩者指向相同 object。

let book = {}
Book.call(book, 'FP in JavaScript', 100)
Object.setPrototypeOf(book, Book.prototype)

或者改用 ES6 的 Object.setPrototypeOf() 亦可。

Book.prototype.showDescription = function() {
  return `${ this.title } / ${ this.price }`
}

也因此我們直接對 Book.prototype 新增 showDescription()，就相當於對 book.__proto__，也就是 book 的 prototype 新增 showDescription()，因為兩者根本指向相同 object。

Class

class Book {
  constructor(title, price) {
    this.title = title
    this.price = price
  }

  showDescription() {
    return `${ this.title } / ${ this.price }`
  }
}

let book = new Book('FP in JavaScript', 100)
book.__proto__ === Book.prototype // ?
book.showDescription() // ?

ES6 支援 class 後，語法又更簡單了，連 prototype 字眼都完全沒看見。

但仍然可看見 book.__proto__ 與 Book.prototype 指向相同 object，再次證明 class 只是 syntatic sugar，ES 依然是 prototype-based，而非 class-based。

Conclusion

  • 隨著時代的進步，ES 寫法不斷精簡，只是黑魔法越來越大，但 ECMAScript 的 prototype-based 本質不變
  • ECMAScript 2015 的 class 寫法雖然完全看不到 prototype，但本質仍是將 method 放在 prototype，是道地的 syntatic sugar
  • Function 自帶 prototype property 且預設為 {}，並不代表 function 的 prototype 為 {}，而是透過 new 之後，將 funciton 的 prototype property 的記憶體位址複製給 object 的 __proto__，使得 object 的 prototype 也是指向 function 的 prototype property 所指向的 object，因此對 function 的 prototype property 動態新增 method，也相當於對 object 的 prototype 新增 method
  • 雖然名義上是針對 constructor function 的 prototype 指定 prototype object，但也相當於是對 __proto__ 指定 prototype object
