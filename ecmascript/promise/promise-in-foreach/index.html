如何在 forEach() 內呼叫 Asynchronous Function ?

2020-03-12

ECMAScript 2015 最大特色是將 Promise 定為語言標準，以 Promise 取代 Callback，當在 forEach() 內使用 Await 時，會有意想不到結果。

Version

macOS Mojave 10.15.3
VS Code 1.43.0
Quokka 1.0.282
ECMAScript 2017

Await in For Loop

let inc = async x => x + 1

let data = [1, 2, 3]

let f = async arr => {
  let sum = 0

  for (let x of arr)
    sum = sum + await inc(x)

  return sum
}

f(data); // ?

第 1 行

let inc = async x => x + 1

ES2017 引進了 async await，只要在 function 名稱前加上 async，則回傳為 promise，而非單純 int。

第 8 行

for (let x of arr)
  sum = sum + await inc(x);

使用 for loop 調用 inc() 累加至 sum。

因為 inc() 為 asynchronous function 回傳 promise，為了等 inc() 回傳值後才繼續執行，特別加上了 await()。

第 5 行

let f = async arr => {
  ...
}

因為 f() 內使用了 await()，ES6 規定在 function 前面要宣告為 async 成為 asynchronous function 回傳 promise。

14 行

f(data) // ?

f() 回傳為 promise。

儘管因為 inc() 宣告為 async， fn() 也成為 asynchronous function，但整體思維與寫法還是與 synchronous function 很像，這就是 async await 的魅力。

Await in forEach()

let inc = async x => x + 1

let data = [1, 2, 3]

let f = arr => {
  let sum = 0
  arr.forEach(async x => sum = sum + await inc(x))
  return sum
}

f(data) // ?

第 7 行

arr.forEach(async x => sum = sum + await inc(x))

隨著 Array.prototype 自帶 forEach() 後，由於可搭配 ES6 的 arrow function，因此實務上越來越多人使用 forEach() 取代 for of loop。

由於 await 是寫在 arrow function 內，因此 arrow function 也要宣告為 async。

但實際執行卻為非預期的 0，why ?

第 8 行

arr.forEach(async x => sum = sum + await inc(x))

Array.prototype.forEach() 為 synchronous function，會一行一行在 stack 執行，這沒問題。

但其 callback 為 asynchronous function，根據 event loop model，該 callback 並不會立即執行，而是先擺到 callback queue 排隊，等全部 synchronous function 都執行完後，才會執行 callback queue 中的 asynchronous function 。

第 10 行

return sum

此為 synchronous 會先執行，但因為 callback quene 中所有的 asynchronous function 皆還未執行，因此 sum 仍然為初始值 0。

Q：所以 await 沒有等待 inc() 的 promise 嗎 ?

await 仍然有等待 inc() 的 Promise()，但問題是在 callback queue 執行 asynchronouns function 時才 await 等待 promise，此時 return sum 早已執行過了，所以 sum 還是 0，因此 await 等待也沒用。

syncForEach()

let inc = async x => x + 1

let data = [1, 2, 3]

Array.prototype.syncForEach = function(f) {
  for(let i = 0; i < this.length; i++)
    f(this[i], i, this)
}

let f = arr => {
  let sum = 0
  arr.syncForEach(async x => sum = sum + await inc(x))
  return sum
};

f(data) // ?

Q：之前這樣解釋還是似懂非懂，可以用 code 解釋嗎 ?

第 5 行

Array.prototype.syncForEach = function(f) {
  for(let i = 0; i < this.length; i++)
    f(this[i], i, this)
}

array.prototype.forEach() 其內部實作相當於 syncForEach()，是封裝 for loop 的 higher order function。

但這裡有個問題，因為 inc() 是 asynchronous function，導致 f() 成為 asynchronous function，但為什麼 f() 之前沒加 await ? 這導致 syncForEach() 成為在 synchronous 去執行 asynchronous f()，所以結果是錯的。

asyncForEach()

let inc = async x => x + 1

let data = [1, 2, 3]

Array.prototype.asyncForEach = async function(f) {
  for(let i = 0; i < this.length; i++)
    await f(this[i], i, this)
}

let f = async arr => {
  let sum = 0
  await arr.asyncForEach(async x => sum = sum + await inc(x))
  return sum
}

f(data); // ?

第 5 行

Array.prototype.asyncForEach = async function(f) {
  for(let i = 0; i < this.length; i++)
    await f(this[i], i, this)
}

自行實作 asyncForEach()，在 f() 前加上 await，也在 function 前加上 async 成為 asynchronous function。

10 行

let f = async arr => {
  let sum = 0
  await arr.asyncForEach(async x => sum = sum + await inc(x))
  return sum
}

如此除了 asyncForEach() 的 callback 要加上 async await 外，連 asyncForEach() 前也要加上 await，最後使 f() 成為 asynchronous function，如此所有的 function 都在 asynchronous 下，不再有 synchronous function 內執行 asynchronous function 問題。

Promise.all()

let inc = async x => x + 1

let data = [1, 2, 3]

let f = arr => Promise
  .all(arr.map(x => inc(x)))
  .then(x => x.reduce((a, x) => a+=x, 0))

f(data) // ?

與其使用 for loop 或 asyncForEach()，其實有更 functional 與 promise 寫法。

改用 map() 使用 inc()，結果為 array 中一堆 pending promise，再使用 Promise.all() 使 array 中的 promise 成為 fulfilled promise。

也由於 Promise.all() 回傳為 promise，可用 then() 直接修改 promise 內部資料，在 then() 使用 reduce() 直接計算其 sum 回傳。

我們可發現整個過程都在 promise 內處理，完全沒透過 await 轉成一般值，也沒透過 imperative 的 for loop 處理，完全使用 FP 的 higher order function 與 pure function 處理，這才是 promise 原本設計。

Ramda

import { pipe, map, sum, andThen } from 'ramda'
import { promiseAll } from 'wink-fp'

let inc = async x => x + 1

let data = [1, 2, 3]

let f = pipe(
  map(inc),
  promiseAll,
  andThen(sum)
)

f(data) // ?

ECMAScript 的 method chaining 在此可讀性不是很好，可改用 Ramda 的 pipe() 組合所有 higher order function。

Wink-fp 的 promiseAll() 為 Promise.all() 的 function 版本，而 Ramda 的 andThen() 則為 Promise.prototype.then() 的 function 版本。 方便 pipeline function composition 使用。

Conclusion

  • 實務上避免在 Array.prototype 下的 method 使用 async await，因為 Array.prototype 下的 method 皆為 synchronous function，也就是 forEach() 其實就是 syncForEach()，會先執行完最後才執行 callback 內的 asynchronous function，導致結果不如預期，此時應該簡單使用 for of loop，就不會有 callback 為 asynchronous function 延後執行問題
  • 結論是要避免在 synchronous function 內執行 asynchronous function，forEach() 本質是 syncForEach()，因此執行 asynchronous inc() 會有問題，除非使用 asyncForEach()，這也是為什麼 ES6 規定只要使用到 await，所有的 function 都要搭配 async 成為 asynchronous function，因為這樣才能避免在 synchronous function 內執行 asynchronous function
  • 若真的搞不懂 syncForEach() 與 asyncForEach() 差異，最少要記得一件事：只能在 for loop 使用 await，不能在 forEach() 內使用 await
  • Promise 原本的設計是希望你用 higher order function + pure function 直接修改 promise 內部值，如此可避開 ECMAScript 複雜的 asynchronous 機制，也不必在 promise 與 await 之中切換
