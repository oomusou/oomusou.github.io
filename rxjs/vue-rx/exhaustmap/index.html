使用 exhaustMap() 每隔一段時間重新讀取 API

2019-10-16

實務上有些資料會不斷變動，因此希望每隔一段時間就重新呼叫 API 讀取資料，這種常見的需求該如何使用 RxJS 實現呢 ?

Version

macOS Catalina 10.15
WebStorm 2019.2.3
Node 10.16.3
Yarn 1.19.1
Vue CLI 3.12.0
Vue 2.16.0
RxJS 6.5.3
Vue-rx 6.2.0
Ramda 0.26.1

Browser

顯示所有書籍的 title 與 price。

Data

{
  "books": [
    {
      "title": "FP in JavaScript",
      "price": 100
    },
    {
      "title": "RxJS in Action",
      "price": 200
    },
    {
      "title": "Speaking JavaScript",
      "price": 300
    }
  ]
}

API 回傳以上 array of object。

API

api/books.js

import axios from 'axios';
import { from } from 'rxjs';

const API = process.env.VUE_APP_API;
const URI = process.env.VUE_APP_BOOKS;

export let fetchBooks$ = () => from(axios.get(`${API}${URI}`));

fetchBooks$() 回傳為 observable。

switchMap() vs. exhaustMap()

<template>
  <div>
    <ul>
      <li v-for="(item, index) in data$" :key="index">
        Title: {{ item.title }}, Price: {{ item.price }}
      </li>
    </ul>
  </div>
</template>

<script>
import { fetchBooks$ } from '@/api/books';
import { interval } from 'rxjs';
import { exhaustMap, pluck } from 'rxjs/operators';

let data$ = interval(1000).pipe(
  exhaustMap(() => fetchBooks$().pipe(
    pluck('data', 'books')
  ))
);

export default {
  name: 'app',
  subscriptions: () => ({
    data$
  }),
}
</script>

第 3 行

<ul>
  <li v-for="(item, index) in data$" :key="index">
    Title: {{ item.title }}, Price: {{ item.price }}
  </li>
</ul>

使用 <li> 與 v-for 顯示所有 data$ 資料。

16 行

let data$ = interval(1000).pipe(
  exhaustMap(() => fetchBooks$().pipe(
    pluck('data', 'books')
  ))
);

想要每 1 秒鐘呼叫一次 API 重新讀取資料。

傳統會使用 setInterval() 搭配 callback 去讀取 API，但 FRP 給了我們另外一種思維。

既然每一秒都要讀取 API，可視為 stream，使用 interval(1000) 產生 observable，再由此 observable 產生另外一個讀取 API 所回傳的 observable。

直覺會使用 switchMap() ：

let data$ = interval(1000).pipe(
  switchMap(() => fetchBooks$().pipe(
    pluck('data', 'books')
  ))
);

switchMap()
一個 observable 的每一個 value 都會對應到另外一個 observable

若注意觀察 swtichMap() 會發現下面的 30 只有兩個，因為新的 50 已經出現，導致最後一個 30 被取消了。

本例 timer(1000) 可視為上面 stream，() => fetchBooks$().pipe(pluck('data', 'books') 則為下面 stream，我們希望不管 interval() 間格為何，每次最少為完整的 stream 顯示，而不要因為有新的 stream 出現，導致原本的 stream 到一半被取消。

exhaustMap() 則不然，儘管有新的 5 已經出現，但原本 30 仍會繼續走完，這才是我們要的。

exhaustMap()
一個 observable 的每一個 value 都會對應到另外一個 observable，且會等前一個 observable 繼續走完，不會中途取消

Extract Function

<template>
  <div>
    <ul>
      <li v-for="(item, index) in data$" :key="index">
        Title: {{ item.title }}, Price: {{ item.price }}
      </li>
    </ul>
  </div>
</template>

<script>
import { fetchBooks$ } from '@/api/books';
import { interval } from 'rxjs';
import { exhaustMap, pluck } from 'rxjs/operators';

let toBooks = () => (
  fetchBooks$().pipe(
    pluck('data', 'books')
  )
);

let data$ = interval(1000).pipe(
  exhaustMap(toBooks)
);

export default {
  name: 'app',
  subscriptions: () => ({
    data$
  }),
}
</script>

22 行

let data$ = interval(1000).pipe(
  exhaustMap(() => fetchBooks$().pipe(
    pluck('data', 'books')
  ))
);

看到 data$ 有兩層 pipe() 似乎有點恐怖，常看到很多人 RxJS 會寫出巢狀 pipe 的 pipe hell，實務上建議一個 function 只有一個 pipe()，若有多的 pipe() 則應各自抽成獨立 function。

16 行

let toBooks = () => (
  fetchBooks$().pipe(
    pluck('data', 'books')
  )
);

將 exhaustMap() 的 callback 抽成獨立的 toBooks()。

22 行

let data$ = interval(1000).pipe(
  exhaustMap(toBooks)
);

如此 data$ 則維持單一 pipe()，可讀性瞬間變高。

Point-free

<template>
  <div>
    <ul>
      <li v-for="(item, index) in data$" :key="index">
        Title: {{ item.title }}, Price: {{ item.price }}
      </li>
    </ul>
  </div>
</template>

<script>
import { fetchBooks$ } from '@/api/books';
import { interval } from 'rxjs';
import { exhaustMap, pluck } from 'rxjs/operators';
import { always } from 'ramda';

let toBooks = always(
  fetchBooks$().pipe(
    pluck('data', 'books')
  )
);

let data$ = interval(1000).pipe(
  exhaustMap(toBooks)
);

export default {
  name: 'app',
  subscriptions: () => ({
    data$
  }),
}
</script>

17 行

let toBooks = always(
  fetchBooks$().pipe(
    pluck('data', 'books')
  )
);

toBooks() 也能使用 Ramda 的 always() 使其 point-free。

Conclusion

  • RxJS 很容易不小心將 callback 寫的太深而不易閱讀，別忘了可將 callback 抽出獨立 function
  • 因為 callback 部分都是 synchronous，別忘了可適時使用 Ramda 使其 point-free 可讀性更高
